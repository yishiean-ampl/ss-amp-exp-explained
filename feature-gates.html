🚪 Feature Gates Deep Dive

Build with Statsig safely: how gates actually evaluate rules, how to read gate metrics without fooling yourself, and how to operate gates without breaking production. Every concept comes with an ELI6 🧒 and an AmpliStore 🛍️ equivalent.

Course Opening

You've shipped behind a gate before. Something probably surprised you. 😅

Rules that didn't fire, metrics showing nothing, a gate behaving weirdly after you raised the rollout %. This course explains why — with three units and lots of clicking.

🧒 ELI6 — what's a feature gate? A feature gate is a light switch for a feature that lives in the cloud instead of in your code. Your app asks "is the switch on for THIS person?" and Statsig answers true or false. You can flip it for everyone, some people, or just your friends — without shipping new code.
🛍️ AmpliStore equivalent Imagine AmpliStore is launching "1-Click Checkout." Instead of releasing it to every shopper at once, you wrap it in a gate: VIP loyalty members get it first, then 25% of everyone else, then 100% once metrics look healthy. Same story you'd tell in the Amplitude demo with an Experiment feature flag in the AmpliStore project — Statsig just makes the gate itself the star of the show.

The through-line for the whole course: gate metrics are signals, not proofs. Keep that in your pocket — it comes back in every unit. 🧵

Unit 1 · ~7 min

⚖️ Rules + Rollout Architecture

Two facts resolve most "my gate isn't behaving" tickets: rules evaluate top-to-bottom with fall-through, and targeting eligibility is a separate control from rollout %.

🔮 Predict first Rule 1 targets iOS users at 20% rollout. A user IS on iOS but falls outside the 20% bucket. Does Statsig evaluate Rule 2 for them?
Yes! Failing a rule for ANY reason — wrong targeting OR outside the rollout % — means the user falls through to the next rule. Play with the simulator below to feel it. 👇

LO1 · Rules evaluate top-to-bottom with fall-through 🪜

A user who fails Rule 1 (didn't match targeting, OR matched but lost the rollout lottery) moves to Rule 2. The decision at each rule is final before the next is considered.

🎮 Simulator: watch a shopper fall through the rules

🧑‍💻 Shopper · bucket hash: —
Rule 1 — "iOS shoppers in the US" · conditions: platform = iOS AND country = US · rollout 100%
Rule 2 — "Everyone else" · condition: everyone · rollout 50%
Default · no rules matched → gate returns false
🧒 ELI6 It's like three doors in a hallway, checked in order. Door 1's bouncer checks his list; if you're not on it (or you lose his coin flip), you walk to Door 2. Nobody ever walks backwards to retry Door 1.

⚠️ The AND vs OR trap

❌ ONE rule, two conditions = AND
Rule 1: platform = iOS + country = US
→ only shoppers who are BOTH iOS AND in the US pass. iOS user in Singapore? Fails. 🙅
✅ TWO rules = OR (via fall-through)
Rule 1: platform = iOS
Rule 2: country = US
→ iOS OR US shoppers pass. Fall-through gives you OR for free. 🎉
🚨 Gotcha This is the #1 gate misconfiguration. Conditions inside a rule are ALWAYS AND. Want OR? Stack separate rules. The UI doesn't make this visually obvious.

LO2 · Targeting and rollout % are separate dials 🎛️

Targeting = who's eligible (the guest list). Rollout % = what fraction of the eligible actually get it (the lottery at the door). Raising % never changes who qualifies — only the sampling rate inside the qualifying pool.

🧒 ELI6 The guest list says "only kids from Room 3 can enter the bouncy castle." The rollout dial says "…and only half of them at a time." Turning the dial doesn't add kids from Room 4 — it just lets more Room 3 kids in.

LO3 · Raising vs lowering % is asymmetric ↕️

🎮 Simulator: 100 shoppers, deterministic hashing

Purple = eligible shoppers. Each has a fixed hash (their lottery number never changes). Move the dial and watch what happens. ⬆️ Raising: everyone green stays green, new greens pop in. ⬇️ Lowering: some greens get kicked out (they flash red) — re-evaluation is real.

✅ Passing: —/100
not eligible eligible, not passing passing 🎉
🚨 Gotcha Lowering rollout % to "remove one specific user" doesn't work — it's probabilistic, they might still pass. To guarantee someone never sees the feature: use an override set to fail, or an explicit exclusion rule.
🛍️ AmpliStore equivalent Gate: one_click_checkout. Rule 1: loyalty_tier = VIP at 100%. Rule 2: everyone else at 25%. A VIP reports "I don't see it!" → first suspect: is loyalty_tier actually being sent on the user object? Missing properties are the #1 cause of "rule doesn't fire" tickets. In your Amplitude AmpliStore demo, this maps to Experiment flag targeting on the UTM Source / user property — same guest-list-vs-dial logic.
🔵 Amplitude comparison Amplitude Experiment flags = close match. Key difference: Statsig's partial-rollout gate automatically generates a pass/fail comparison panel (gate-as-test). In Amplitude, flag and experiment are more separate objects.

Unit 2 · ~8 min

📊 Reading Gate Metrics + Diagnostics

The most misunderstood part of gates. Two facts fix almost all confusion: metrics are per-rule, and lift needs a comparison group to exist.

🔮 Predict first Your gate has Rule 1 (internal users, 100%) and Rule 2 (everyone else, 50%), live for 2 weeks. How many lift calculations appear?
One — Rule 2 only. Rule 1 at 100% has nobody failing, so there's nothing to compare against. No fail group = no lift. See it live below. 👇

LO4 · Metrics are per-rule — no aggregate view 🧩

🎮 Simulator: drag Rule 2's rollout and watch the lift panel

Rule 1 · Internal users · 100% 🔒
😶 No lift shown.
Everyone passes → no comparison group.
Not broken — by design.
Rule 2 · Everyone else · 50%
🧒 ELI6 You can't tell if the new vitamin works if every single kid takes it. You need some kids who didn't, to compare. A rule at 100% = every kid took the vitamin = 🤷 no comparison possible.
🚨 Gotcha "My gate is at 50% but metrics show nothing!" — the panel appears, but lift only shows for metrics where exposed users also fired the metric event. No purchase events in the pass group yet? Empty panel. Empty ≠ broken.

LO5 · Exposure logs STOP at 0% and 100% — by design 📴

🎮 Simulator: the exposure event stream

🧒 ELI6 The referee only keeps score while there are two teams playing. When one team has everybody (100%) or nobody (0%), there's no game to score, so the referee puts the whistle down. Silence = the game ended, not a broken whistle.
🚨 Gotcha "No exposures landing in BigQuery — the pipeline is broken!" → Check the rollout % first. At 0% or 100%, warehouse exports stop receiving exposure events. Expected behavior, not an incident.

LO6 · Diagnostics tab = first stop when a gate "isn't working" 🩺

Three checks, in order: ① Is the gate being evaluated at all (non-zero exposure count)? ② Does the pass % match your expected rollout %? ③ Are passing users also firing metric events?

🛍️ AmpliStore equivalent one_click_checkout Rule 2 at 50%: the lift panel compares purchase rate and cart abandonment between pass and fail shoppers. If nobody's completed a checkout yet today, the panel is empty — wait for events, don't file a ticket. In the AmpliStore Amplitude project, the analogous debug motion is checking exposure events in the Experiment diagnostics view before touching anything else.
🔵 Amplitude comparison Amplitude Experiment has a Diagnostics view (partial match). Difference: Statsig shows lift per rule inline on the gate; Amplitude separates flag from experiment, so "flag metrics" doesn't map 1:1.

⭐ The big idea: gate metrics are observational, not causal

🧒 ELI6 If you give cookies only to the kids who finished their homework, and then notice "cookie kids are better at math!" … it's not the cookies. 🍪 You picked the good-at-homework kids. Gate targeting picks people by a rule, not randomly — so differences might come from WHO you picked, not WHAT you gave them. Random picking (an experiment) is what makes it fair.

Unit 3 · ~8 min

🛡️ Safe Operations: Overrides, Archiving, Graduation

Three things cause unintended production changes: override permissions, archiving behavior, and skipping the gate-to-experiment conversion.

🔮 Predict first You archive a gate that's been at 100% for three months. Do users still see the feature afterwards?
No! 💥 Archiving removes the gate from the SDK payload, so checkGate() returns false — the feature vanishes. Step through the incident below. 👇

LO8 · Archiving removes the gate from the SDK payload 🗑️

🎬 Animation: "just cleaning up" — a production incident in 4 steps

🛍️ AmpliStore
⚡ 1-Click Checkout
// SDK config payload { "gates": { "one_click_checkout": { "state": "Launched", "rollout": 100 } } }
Step 1/4 — Gate is Launched at 100%. Shoppers happily 1-click away. 😌
🧒 ELI6 The gate is a sticky note on the fridge that says "dessert is allowed." Your code checks the fridge every morning. Archiving throws away the sticky note — and when the code finds no note, it assumes "no dessert" (false). If you never removed the "check the fridge" habit from the code, dessert disappears for everyone. 🍰❌
🚨 Gotcha — the correct cleanup order ① Remove the checkGate() call from code → ② merge and deploy → ③ THEN archive the gate. Archiving first = users lose the feature instantly. (Or leave it in "Launched" state indefinitely until code cleanup happens; "Permanent" gates even resolve true after archiving.)

LO7 · Overrides need LAUNCH permission, not edit 🔐

Overrides bypass targeting rules and hit production users — so saving one requires launch permission. Edit-only users can configure an override in the UI but the save fails.

🧒 ELI6 Anyone in class can write on the whiteboard plan for the field trip (edit). But only the teacher can open the bus door (launch). An override is opening the door for a specific kid — teacher-level move.
🚨 Gotcha QA engineer with edit access sets an override, clicks save → permission error. Fix: request launch access, or have an admin enable "Bypass Reviews for Overrides" at the project level to streamline QA.

LO9 · Graduate the gate to an experiment for causal proof 🎓

Want to know if the feature caused the metric movement? Convert the gate to an experiment — a first-class UI action in Statsig. Targeting logic is preserved; you gain hypothesis framing, named variants, and full Pulse with statistical testing.

🛍️ AmpliStore equivalent one_click_checkout looks great at 25% — purchase rate up. Before shouting "ship it!" in the AmpliStore QBR: convert to an experiment, randomize 50/50, and let Pulse tell you whether the lift is real or just VIP-selection bias. In the Amplitude demo the same story = create an experiment referencing the existing flag (the conversion is less of a one-click motion there — that's the demo contrast).
🔵 Amplitude comparison Amplitude's flag-to-experiment flow = close match. Difference: Statsig's conversion is a first-class UI action; in Amplitude it looks more like creating a new experiment that references the existing flag.
